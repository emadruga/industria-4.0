# **Questão 1 - Confiança em processos e sistemas de informação**

| Como a liderança e as equipes utilizam e confiam em processos estruturados e sistemas de informação para apoiar suas decisões e operações diárias? |  |
| :---- | :---- |
| ***Capacidade em medição:*** | Confiança em processos e sistemas de informação |

| *Respostas* |  |
| :---: | :---: |
| ***Peso*** | ***Item de resposta*** |
| 0 | Não há processos estruturados nem sistemas de informação digitais; decisões e operações baseiam-se exclusivamente em experiência pessoal e intuição. |
| 1 | Existem processos básicos documentados, mas as informações são gerenciadas manualmente (planilhas locais, registros em papel); há baixa confiança nos dados disponíveis. |
| 2 | Sistemas digitais básicos estão implementados para coleta de dados, mas são subutilizados; equipes ainda dependem parcialmente de métodos manuais e há desconfiança ocasional na precisão dos sistemas. |
| 3 | Processos e sistemas digitais são utilizados regularmente para monitoramento operacional; a maioria das decisões rotineiras é apoiada por dados dos sistemas, com confiança crescente na qualidade das informações. |
| 4 | Processos estão plenamente integrados com sistemas de informação; dados são usados de forma sistemática e confiável para análise de causas, tomada de decisões e identificação de melhorias; há alta transparência e rastreabilidade. |
| 5 | Sistemas de informação são preditivos e integrados; liderança e equipes confiam plenamente em análises avançadas, simulações e alertas automatizados para antecipar problemas e otimizar processos. |
| 6 | Processos e sistemas são totalmente autônomos e adaptativos; decisões operacionais são tomadas automaticamente com base em dados em tempo real; liderança atua como facilitadora e supervisora estratégica. |

| *Possíveis fontes de evidências:* |
| :---- |
| **A) Artefatos e onde buscar**
Manuais de processos e procedimentos operacionais padrão (POPs) disponíveis digitalmente.
Registros de uso de sistemas (logs de acesso, frequência de consultas, painéis ativos).
Documentação de sistemas de informação (ERPs, MES, dashboards, catálogos de dados).
Atas de reuniões operacionais que referenciam dados e relatórios de sistemas.
Evidências de integração entre sistemas (APIs, fluxos de dados automatizados).
**B) Métricas/KPIs**
% de decisões operacionais baseadas em dados de sistemas versus intuição ou informações manuais.
Taxa de adoção de sistemas digitais por área/função (usuários ativos versus total de colaboradores).
Tempo médio entre coleta de dados e disponibilização em sistemas (latência de dados).
% de processos críticos com rastreabilidade digital completa.
Taxa de confiança declarada em pesquisas internas sobre qualidade dos dados e sistemas.
Frequência de uso de relatórios e dashboards para decisões (acessos/semana).
**C) Sinais por nível**
**N0:** ausência de sistemas digitais; registros manuais ou inexistentes; decisões baseadas exclusivamente em experiência.
**N1:** processos documentados em papel ou planilhas locais; dados fragmentados e pouco compartilhados; baixa confiança nas informações.
**N2:** sistemas básicos implementados (ex.: ERP inicial); uso limitado; equipes ainda preferem métodos tradicionais; desconfiança ocasional nos dados.
**N3:** uso regular de sistemas para monitoramento; dados descritivos são acessados frequentemente; decisões rotineiras apoiadas por relatórios; confiança crescente.
**N4:** integração de sistemas; dados confiáveis e rastreáveis; análises de causa-raiz e KPIs sistemáticos; alta transparência; processos digitais bem estabelecidos.
**N5:** sistemas preditivos e analíticos avançados; uso de simulações e cenários; alertas automatizados para antecipação de problemas; confiança plena em dados.
**N6:** processos autônomos com ajustes em tempo real; decisões automatizadas; sistemas adaptativos; liderança focada em estratégia e supervisão.
**D) Amostragem**
Analisar 5-7 processos críticos (ex.: planejamento de produção, manutenção preventiva, gestão de estoques, controle de qualidade) e verificar: (1) se há sistemas digitais suportando o processo, (2) nível de uso efetivo pelos colaboradores, (3) evidências de confiança nas informações (referências em atas, decisões documentadas), (4) grau de automação e integração.
Realizar entrevistas ou pesquisas com líderes e operadores sobre confiança nos sistemas e processos. |

# **Questão 2 - Segurança da informação e consciência de riscos cibernéticos**

| Como a organização desenvolve e mantém a consciência de segurança cibernética e a cultura de proteção de informações entre todos os colaboradores? |  |
| :---- | :---- |
| ***Capacidade em medição:*** | Confiança em processos e sistemas de informação |

| *Respostas* |  |
| :---: | :---: |
| ***Peso*** | ***Item de resposta*** |
| 0 | Não há programa de conscientização sobre segurança; colaboradores desconhecem riscos cibernéticos e políticas de segurança da informação. |
| 1 | Existe programa básico de segurança para atender requisitos mínimos de compliance; treinamento pontual e reativo focado apenas em cumprir auditoria. |
| 2 | Programa de conscientização limitado com treinamento anual básico; colaboradores têm conhecimento superficial sobre segurança mas não aplicam consistentemente na prática. |
| 3 | Programa estruturado de conscientização com treinamentos regulares; equipes compreendem principais riscos e seguem políticas de segurança nas operações rotineiras. |
| 4 | Programa proativo de segurança com múltiplas ações de reforço ao longo do ano; colaboradores demonstram comportamentos seguros consistentes e reportam incidentes; há medição de mudança comportamental. |
| 5 | Cultura de segurança consolidada como parte do dia a dia organizacional; colaboradores são vigilantes, compartilham conhecimento sobre ameaças e participam ativamente da melhoria contínua; programa baseado em métricas e evolução contínua. |
| 6 | Segurança é valor cultural profundamente enraizado; colaboradores são embaixadores da segurança; organização utiliza tecnologias avançadas (IA, machine learning) para detecção e resposta automatizada; melhoria contínua com métricas sofisticadas de impacto. |

| *Possíveis fontes de evidências:* |
| :---- |
| **A) Artefatos e onde buscar**
Programa de Security Awareness documentado com cronograma, temas e público-alvo.
Materiais de treinamento (cursos, vídeos, newsletters, campanhas, simulações de phishing).
Políticas de segurança da informação publicadas e acessíveis a todos colaboradores.
Certificados de conclusão de treinamentos e taxa de participação.
Registros de simulações de phishing e taxa de sucesso/cliques.
Relatórios de incidentes de segurança reportados por colaboradores.
Planos de resposta a incidentes (incident response playbooks).
Evidências de programa de reconhecimento para comportamentos seguros.
**B) Métricas/KPIs**
Taxa de conclusão de treinamentos obrigatórios de segurança por área/função.
Taxa de cliques em simulações de phishing ao longo do tempo.
Taxa de reporte de e-mails/atividades suspeitas pelos colaboradores.
Número de incidentes de segurança causados por erro humano.
Tempo médio para detecção e resposta a incidentes (MTTD/MTTR).
Pontuação em avaliações de conhecimento sobre segurança.
% de colaboradores que completam treinamentos avançados/especializados.
Frequência de comunicações e campanhas de conscientização (newsletters, alertas).
**C) Sinais por nível**
**N0:** ausência de programa; colaboradores desconhecem políticas; incidentes frequentes por desconhecimento.
**N1:** treinamento pontual apenas para compliance; pouco engajamento; foco em "cumprir tabela".
**N2:** treinamento anual genérico; conhecimento superficial; aplicação inconsistente na prática.
**N3:** programa estruturado; treinamentos regulares; políticas conhecidas e seguidas; reportes ocasionais de suspeitas.
**N4:** programa proativo com múltiplas ações; comportamento seguro consistente; medição de mudança comportamental; cultura de reporte.
**N5:** segurança incorporada à cultura; vigilância ativa; compartilhamento de conhecimento; melhoria contínua baseada em métricas.
**N6:** segurança como valor cultural; embaixadores internos; uso de IA para detecção; resposta automatizada; métricas sofisticadas de impacto comportamental e redução de risco.
**D) Amostragem**
Analisar últimos 6-12 meses do programa de Security Awareness: cronograma executado, taxa de participação, resultados de simulações de phishing em 3-5 campanhas, taxa de reporte de incidentes.
Realizar entrevistas com amostra de colaboradores de diferentes áreas para avaliar conhecimento sobre políticas e comportamentos de segurança.
Analisar 5-10 incidentes recentes para verificar se foram causados por falha humana e se havia treinamento adequado. |

# **Questão 3 - Interoperabilidade e integração de sistemas**

| Qual o grau de integração e interoperabilidade entre os diversos sistemas de informação utilizados pela organização (internos e com parceiros externos)? |  |
| :---- | :---- |
| ***Capacidade em medição:*** | Confiança em processos e sistemas de informação |

| *Respostas* |  |
| :---: | :---: |
| ***Peso*** | ***Item de resposta*** |
| 0 | Sistemas operam completamente isolados (silos); não há compartilhamento de dados entre sistemas; integração inexistente. |
| 1 | Integração mínima e manual entre sistemas críticos através de transferência manual de arquivos ou reentrada de dados; processos lentos e sujeitos a erros. |
| 2 | Algumas integrações pontuais automatizadas entre sistemas principais; dados fluem parcialmente mas ainda há necessidade de intervenção manual frequente e reconciliação. |
| 3 | Sistemas principais integrados através de interfaces e APIs; dados fluem entre sistemas de forma estruturada mas ainda existem gaps e integrações que requerem desenvolvimento customizado. |
| 4 | Integração abrangente horizontal (entre áreas) e vertical (do chão de fábrica ao ERP); arquitetura de integração bem definida; dados fluem consistentemente com rastreabilidade; integrações com principais parceiros externos. |
| 5 | Plataforma de integração empresarial consolidada; arquitetura orientada a serviços (SOA) ou microserviços; integração em tempo real; ecossistema digital integrado incluindo parceiros, fornecedores e clientes. |
| 6 | Arquitetura totalmente interoperável e adaptativa; integração nativa e automática de novos sistemas; uso de padrões abertos; ecossistema digital dinâmico com IoT, edge computing e cloud integrados; sincronização em tempo real com toda cadeia de valor. |

| *Possíveis fontes de evidências:* |
| :---- |
| **A) Artefatos e onde buscar**
Arquitetura de integração documentada (diagramas de integração, mapas de fluxo de dados).
Inventário de sistemas e aplicações com matriz de integrações.
Documentação de APIs, web services e interfaces de integração.
Plataforma de integração empresarial (ESB, iPaaS) se existir.
Logs de integração e monitoramento de fluxos de dados.
Contratos e acordos de integração com parceiros externos (EDI, portais B2B).
Políticas de governança de integração e padrões técnicos.
Roadmap de integração e projetos de integração em andamento.
**B) Métricas/KPIs**
% de sistemas críticos integrados versus isolados.
Número total de integrações automatizadas ativas.
Tempo médio para implementar nova integração.
Latência média de sincronização de dados entre sistemas.
Taxa de falhas em integrações (disponibilidade das integrações).
% de processos de negócio que dependem de dados integrados.
Volume de transações/dados trafegando via integrações.
% de entrada manual de dados versus captura automatizada.
Nível de padronização de integrações (uso de APIs padrão vs. customizações).
**C) Sinais por nível**
**N0:** sistemas completamente isolados; silos de informação; ausência total de integração.
**N1:** transferências manuais de arquivos; reentrada de dados; reconciliação manual frequente; erros comuns.
**N2:** algumas integrações automatizadas pontuais; ainda significativa intervenção manual; dados parcialmente sincronizados.
**N3:** sistemas principais integrados via APIs; fluxo estruturado mas ainda gaps; algumas customizações necessárias.
**N4:** integração abrangente horizontal e vertical; arquitetura definida; dados fluem consistentemente; parceiros externos integrados.
**N5:** plataforma de integração consolidada; SOA/microserviços; tempo real; ecossistema digital amplo incluindo cadeia de valor.
**N6:** arquitetura totalmente interoperável; integração automática de novos sistemas; padrões abertos; IoT e cloud nativos; sincronização em tempo real end-to-end.
**D) Amostragem**
Mapear arquitetura de integração atual: listar todos os sistemas principais, identificar quais estão integrados e como (manual, batch, tempo real).
Selecionar 3-5 processos de negócio críticos (ex.: order-to-cash, procure-to-pay, plan-to-produce) e avaliar nível de integração dos sistemas envolvidos.
Verificar documentação técnica de integrações com parceiros externos chave (fornecedores, clientes, logistics). |

# **Questão 4 - Disponibilidade e confiabilidade de sistemas críticos**

| Qual o nível de disponibilidade, confiabilidade e resiliência dos sistemas de informação críticos para as operações da organização? |  |
| :---- | :---- |
| ***Capacidade em medição:*** | Confiança em processos e sistemas de informação |

| *Respostas* |  |
| :---: | :---: |
| ***Peso*** | ***Item de resposta*** |
| 0 | Sistemas críticos frequentemente indisponíveis; falhas recorrentes sem planos de contingência; operação severamente impactada por instabilidade tecnológica. |
| 1 | Sistemas apresentam disponibilidade irregular; incidentes frequentes com resolução reativa e demorada; ausência de acordos de nível de serviço (SLAs) ou monitoramento estruturado. |
| 2 | Disponibilidade básica com interrupções ocasionais; existe monitoramento mas resposta ainda é reativa; SLAs definidos mas nem sempre cumpridos; backups básicos implementados. |
| 3 | Sistemas críticos com boa disponibilidade (>95%); monitoramento proativo; SLAs estabelecidos e geralmente cumpridos; planos de recuperação de desastres documentados e testados ocasionalmente. |
| 4 | Alta disponibilidade (>99%); infraestrutura redundante para sistemas críticos; monitoramento em tempo real com alertas automatizados; SLAs rigorosos cumpridos consistentemente; testes regulares de DR. |
| 5 | Disponibilidade muito alta (>99,5%); arquitetura resiliente com failover automático; self-healing capabilities; monitoramento preditivo que antecipa falhas; recuperação rápida (RTO/RPO baixos). |
| 6 | Disponibilidade próxima a 100%; arquitetura distribuída e altamente resiliente; sistemas adaptativos que se auto-otimizam; zero downtime para manutenções; recuperação instantânea; capacidade de operar em modo degradado mantendo funções essenciais. |

| *Possíveis fontes de evidências:* |
| :---- |
| **A) Artefatos e onde buscar**
Inventário de sistemas críticos classificados por impacto no negócio (análise de impacto de negócio - BIA).
SLAs (Service Level Agreements) documentados para cada sistema crítico.
Relatórios de disponibilidade e uptime de sistemas.
Logs de incidentes e problemas em ferramentas de ITSM (ServiceNow, JIRA Service Management).
Arquitetura de infraestrutura mostrando redundâncias, clusters, load balancers.
Plano de Continuidade de Negócios (BCP) e Plano de Recuperação de Desastres (DRP).
Relatórios de testes de DR e simulações de falha.
Documentação de procedimentos de backup e restore com RPO/RTO definidos.
Dashboards de monitoramento de infraestrutura e aplicações (APM tools).
**B) Métricas/KPIs**
Uptime/disponibilidade dos sistemas críticos (%).
MTBF (Mean Time Between Failures) - tempo médio entre falhas.
MTTR (Mean Time To Repair) - tempo médio para reparo.
MTTD (Mean Time To Detect) - tempo médio para detectar incidentes.
Número de incidentes críticos (P1/P2) por mês.
% de SLAs cumpridos.
RPO (Recovery Point Objective) e RTO (Recovery Time Objective) reais versus meta.
Taxa de sucesso em testes de DR.
Frequência de manutenções não planejadas versus planejadas.
**C) Sinais por nível**
**N0:** sistemas frequentemente indisponíveis; falhas recorrentes; ausência de planos de contingência; operação severamente impactada.
**N1:** disponibilidade irregular; incidentes frequentes; resposta reativa e lenta; sem SLAs formais ou monitoramento estruturado.
**N2:** disponibilidade básica; interrupções ocasionais; monitoramento implementado mas reativo; SLAs definidos mas cumprimento irregular; backups básicos.
**N3:** boa disponibilidade (>95%); monitoramento proativo; SLAs geralmente cumpridos; DRP documentado e testado ocasionalmente.
**N4:** alta disponibilidade (>99%); infraestrutura redundante; monitoramento em tempo real; SLAs rigorosos cumpridos; testes regulares de DR.
**N5:** disponibilidade muito alta (>99,5%); arquitetura resiliente com failover automático; self-healing; monitoramento preditivo; recuperação rápida.
**N6:** disponibilidade próxima a 100%; arquitetura distribuída resiliente; auto-otimização; zero downtime; recuperação instantânea; modo degradado funcional.
**D) Amostragem**
Analisar disponibilidade dos 5-10 sistemas mais críticos nos últimos 6-12 meses: tempo de uptime, número e duração de incidentes.
Revisar últimos 5 incidentes críticos: tempo de detecção, diagnóstico e resolução; causa raiz; impacto no negócio.
Verificar última execução de teste de DR: quando foi, sistemas testados, resultados, gaps identificados.
Entrevistar equipes de operações sobre confiabilidade percebida dos sistemas. |

| *Glossário* |
| ----- |
| **Processos estruturados:** Conjunto documentado e padronizado de atividades e fluxos de trabalho que orientam a execução de tarefas de forma consistente e eficiente. |
| **Sistemas de informação:** Ferramentas digitais (softwares, plataformas, sistemas integrados) que coletam, armazenam, processam e disponibilizam dados para apoiar operações e decisões. Exemplos: ERP, MES, SCADA, dashboards, BI. |
| **Confiança em dados:** Grau de segurança e credibilidade que líderes e equipes depositam na qualidade, precisão, atualidade e integridade das informações fornecidas pelos sistemas. |
| **Rastreabilidade digital:** Capacidade de acompanhar e registrar digitalmente todas as etapas de um processo, desde a origem até o resultado final, garantindo transparência e auditabilidade. |
| **Sistemas preditivos:** Ferramentas que utilizam modelos analíticos e históricos de dados para antecipar eventos futuros, como falhas de equipamentos, variações de demanda ou desvios de qualidade. |
| **Autonomia de sistemas:** Capacidade de sistemas de informação executarem decisões operacionais automaticamente, com base em regras pré-definidas e dados em tempo real, minimizando a necessidade de intervenção humana. |
| **MES (Manufacturing Execution System):** Sistema de execução de manufatura que monitora e controla processos produtivos em tempo real. |
| **ERP (Enterprise Resource Planning):** Sistema integrado de gestão empresarial que centraliza informações de diferentes áreas (finanças, operações, RH, etc.). |
| **SCADA (Supervisory Control and Data Acquisition):** Sistema de supervisão e aquisição de dados utilizado para monitoramento e controle de processos industriais. |
| **Security Awareness:** Programa de conscientização sobre segurança da informação que visa educar colaboradores sobre riscos cibernéticos, políticas de segurança e comportamentos seguros. |
| **Phishing:** Técnica de ataque cibernético que utiliza e-mails ou mensagens fraudulentas para enganar usuários e obter informações confidenciais ou acesso a sistemas. |
| **Cultura de segurança:** Conjunto de valores, crenças e comportamentos compartilhados em uma organização que priorizam a proteção de informações e a prevenção de riscos cibernéticos. |
| **Incident Response:** Processo estruturado de detecção, análise, contenção, erradicação e recuperação de incidentes de segurança da informação. |
| **MTTD (Mean Time To Detect):** Tempo médio necessário para detectar um incidente de segurança ou falha em sistemas. |
| **MTTR (Mean Time To Repair):** Tempo médio necessário para reparar e restaurar um sistema ou serviço após uma falha. |
| **Interoperabilidade:** Capacidade de diferentes sistemas, dispositivos ou organizações trabalharem juntos e trocarem informações de forma eficaz. |
| **API (Application Programming Interface):** Interface de programação que permite a comunicação e troca de dados entre diferentes sistemas e aplicações. |
| **ESB (Enterprise Service Bus):** Plataforma de integração que facilita a comunicação entre diferentes aplicações e sistemas empresariais. |
| **SOA (Service-Oriented Architecture):** Arquitetura de software que organiza funcionalidades em serviços independentes e reutilizáveis que podem ser acessados via rede. |
| **EDI (Electronic Data Interchange):** Troca eletrônica estruturada de documentos comerciais entre sistemas de diferentes organizações. |
| **IoT (Internet of Things):** Rede de dispositivos físicos conectados à internet que coletam e trocam dados automaticamente. |
| **SLA (Service Level Agreement):** Acordo formal que define níveis esperados de serviço, incluindo disponibilidade, desempenho e tempo de resposta. |
| **Uptime:** Percentual de tempo em que um sistema está disponível e operacional. |
| **Disponibilidade:** Capacidade de um sistema estar acessível e funcionando quando necessário, geralmente expressa em porcentagem. |
| **Redundância:** Duplicação de componentes críticos de infraestrutura para garantir continuidade em caso de falha. |
| **Failover:** Capacidade de um sistema transferir automaticamente operações para um sistema backup redundante em caso de falha. |
| **Self-healing:** Capacidade de sistemas detectarem e corrigirem problemas automaticamente sem intervenção humana. |
| **BCP (Business Continuity Plan):** Plano que define procedimentos para manter operações críticas durante e após uma interrupção. |
| **DRP (Disaster Recovery Plan):** Plano específico para recuperar sistemas de TI e dados após um desastre ou interrupção grave. |
| **RPO (Recovery Point Objective):** Quantidade máxima aceitável de perda de dados medida em tempo (ex.: última 1 hora de dados). |
| **RTO (Recovery Time Objective):** Tempo máximo aceitável para restaurar um sistema ou serviço após uma interrupção. |
| **MTBF (Mean Time Between Failures):** Tempo médio entre falhas consecutivas de um sistema ou componente. |
